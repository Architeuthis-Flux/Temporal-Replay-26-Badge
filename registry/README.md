# Temporal Badge — Community Apps Registry

This folder hosts the registry the badge fetches over WiFi to populate
its **Community Apps** screen (formerly "Asset Library"). The registry
lets us ship installable apps + downloadable user files (the DOOM WAD,
sound packs, fonts, etc.) without rebuilding the firmware.

Two files live here:

- [`community_apps.json`](community_apps.json) — **schema v2**, what
  current firmware reads. Auto-generated by
  [`firmware/scripts/generate_startup_files.py`](../firmware/scripts/generate_startup_files.py)
  from `firmware/data/`. Don't edit by hand; edit the source files and
  re-run the generator.
- [`registry.json`](registry.json) — **schema v1**, hand-edited. Kept
  for backwards compatibility so badges in the field that haven't
  firmware-updated still see their original asset list.

> See [`firmware/docs/STORAGE-MODEL.md`](../firmware/docs/STORAGE-MODEL.md)
> for the full picture of where things live on the badge and what
> survives each kind of flash.

---

## How the badge consumes it

1. The user (or the firmware build) sets `community_apps_url` in
   `firmware/settings.txt`. The default points at this folder via
   `raw.githubusercontent.com/.../main/registry/community_apps.json`.
2. Once a day, when WiFi is connected, the badge fetches the URL and
   shows entries in `Settings → Community Apps`.
3. The user picks an entry; the badge downloads the file(s) into a
   `.tmp` path on FATFS, optionally SHA-256 verifies, and atomically
   renames into place.

The legacy settings key `asset_registry_url` still works (the
firmware reads both names into the same field).

## Schema v2 (`community_apps.json`)

Two kinds of entries: single files (`kind: "file"`) and multi-file
apps (`kind: "app"`).

```json
{
  "schema_version": 2,
  "generator": "generate_startup_files.py",
  "assets": [
    {
      "id": "doom1-shareware",
      "kind": "file",
      "name": "DOOM 1 Shareware WAD",
      "version": "1.9",
      "dest_path": "/doom1.wad",
      "url": "https://github.com/.../doom1.wad",
      "sha256": "<64 hex>",
      "size": 4196020,
      "min_free_bytes": 4500000,
      "description": "DOOM 1 shareware levels."
    },
    {
      "id": "tardigotchi",
      "kind": "app",
      "name": "Tardigotchi",
      "version": "8b139b84ef0b",
      "dest_dir": "/apps/tardigotchi",
      "size": 33744,
      "min_free_bytes": 37840,
      "description": "Hatch and care for a tiny tardigrade.",
      "files": [
        {"path": "/main.py",   "size": 187,   "sha256": "...", "url": "https://..."},
        {"path": "/engine.py", "size": 24006, "sha256": "...", "url": "https://..."},
        {"path": "/icon.py",   "size": 9551,  "sha256": "...", "url": "https://..."}
      ]
    }
  ]
}
```

### File entry (`kind: "file"`)

| Field | Notes |
|-------|-------|
| `id` | Stable NVS key (≤ 32 chars). Treat like a primary key — never reuse. |
| `name` | Display name shown on the OLED. |
| `version` | Opaque string. Any change vs. the installed value flips the entry to "Update available". |
| `dest_path` | Absolute FATFS path. Atomic-renamed from `<dest_path>.tmp` after download. |
| `url` | HTTPS download URL. Redirects (302) are followed. TLS is unverified — see security note below. |
| `sha256` | Optional 64-char hex. **Corruption check only**, not a signature. |
| `size` | Bytes. Recommended; lets the badge show a progress bar. |
| `min_free_bytes` | Optional. Refuse install if FATFS has less free space. |
| `description` | Short blurb shown in the detail screen. |

### App entry (`kind: "app"`)

| Field | Notes |
|-------|-------|
| `id`, `name`, `version`, `description` | As above. `version` is auto-derived from `sha256(concat of file sha256s)` so any change to any file in the bundle bumps the version. |
| `dest_dir` | Absolute FATFS directory the app's files land in. |
| `size`, `min_free_bytes` | Aggregate of all files in the bundle. |
| `files` | Array of `{path, size, sha256, url}` objects, one per file in the bundle. `path` is relative to `dest_dir` (e.g. `"/main.py"` lands at `<dest_dir>/main.py`). |

There is **no separate per-app `manifest.json`** — the file list is
inlined into the registry so a single HTTP fetch covers the whole
catalog. Older revisions of this schema referenced an external
`manifest_url`; that field is gone.

**Note**: the multi-file install path on the badge is currently a
follow-up — today's `AssetRegistry::install()` handles `kind:"file"`
entries and treats `kind:"app"` as a no-op until the multi-file
installer lands. See `firmware/docs/OTA-MAINTAINER.md`.

## Where the asset bytes live

For files too big to bake into firmware (≳ 100 KB — DOOM WAD, sound
packs, alternate fonts), the canonical home is **a GitHub Release
attachment** managed by
[`release-assets/manifest.json`](../release-assets/manifest.json).
Use the stable `https://github.com/<owner>/<repo>/releases/latest/download/<name>`
URL — it survives across firmware releases so cached registries stay
valid indefinitely.

For everything else (Python apps, small assets), point `url` /
`manifest_url` at any HTTPS host you control. The default firmware
points at `cdn.jsdelivr.net/gh/...` which is a free, fast CDN over
the GitHub repo — no infrastructure to maintain.

### Hosting your own registry

Any HTTPS URL serving valid JSON works:

- **GitHub raw** — `https://raw.githubusercontent.com/…/main/registry/community_apps.json` (firmware default).
- **GitHub Pages** — drop a `community_apps.json` in your `gh-pages` branch.
- **jsDelivr** — works for repos under their free 50 MB per-package limit; the Temporal Replay 26 repo is currently over the limit (submodules + DOOM WAD + animations push it past 50 MB) so jsDelivr replies `403 Package size exceeded`. Stick with raw or self-host.
- **Cloudflare R2 / Pages / Workers** — see `firmware/docs/OTA-MAINTAINER.md`.
- **Self-hosted** — any plain `nginx` or `python -m http.server` works.

The badge sends `User-Agent: TemporalBadge-OTA/1.0` if you want to
filter access logs.

## Adding a new entry

For a single file (DOOM WAD, sound pack, etc.):

1. Edit
   [`firmware/scripts/generate_startup_files.py`](../firmware/scripts/generate_startup_files.py)
   if the file lives outside `firmware/data/{docs,images,messages}/`,
   or just drop the file into the right `firmware/data/` subdirectory
   for the auto-emit path.
2. Run `python3 firmware/scripts/generate_startup_files.py`.
3. Commit `registry/community_apps.json` along with the source file.

For a multi-file app:

1. Drop your app folder into `firmware/data/apps/<your_app>/`.
2. Run the generator. It emits `firmware/data/apps/<your_app>/manifest.json`
   and adds the entry to `community_apps.json`.
3. Commit everything.

There's no validation step on merge — keep the JSON well-formed and
don't reuse an existing `id`.

## Security stance

This is open-source firmware with no secrets to protect:

- TLS uses `WiFiClientSecure::setInsecure()` — no certificate pinning.
- `sha256` fields are corruption checks, not signatures.
- There's no image signing for assets.

The threat model is "don't brick the badge", which is mitigated by:

- a battery ≥ 30 % guard for firmware updates (unless USB is plugged),
- atomic `<dest_path>.tmp` → `<dest_path>` rename for asset writes,
- bootloader rollback for firmware OTA failures.

If your fleet needs a hardened registry, fork `firmware/src/ota/` and
bring your own keys.

## Schema v1 (legacy)

The previous schema is documented inline at the top of
[`registry.json`](registry.json) and was structurally identical to v2
`kind:"file"` entries (no `kind` field, no `dest_dir`/`manifest_url`,
no app-bundle support). Older firmware out in the field reads it
directly. New entries should go into `community_apps.json` — leave
v1 frozen.
