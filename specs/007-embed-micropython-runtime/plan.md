# Implementation Plan: Embed MicroPython Scripting Runtime

**Branch**: `007-embed-micropython-runtime` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/007-embed-micropython-runtime/spec.md`

---

## Summary

Embed MicroPython v1.27.0 as a scripting runtime inside the Arduino C++ firmware (Firmware-0308-modular). The C++ firmware remains the host OS. MicroPython is initialized once during the boot sequence (WiFi connection window), runs inline in `loop()` on Core 1 when a Python app is selected from the menu, and is soft-reset between app launches.

The runtime exposes a single `badge` C extension module covering display (namespaced sub-object), button input, IR send/receive, WiFi/HTTP, GPIO/ADC, GC control, and app exit. Apps are plain `.py` files on the badge VFS. No existing firmware behavior changes.

---

## Technical Context

**Language/Version**: Arduino C++ (ESP32 Arduino core 3.x) + MicroPython v1.27.0 embed port
**Primary Dependencies**: MicroPython `ports/embed` (generated C sources); existing libraries unchanged
**Storage**: Badge VFS (FAT); apps as `/apps/<name>.py`; `micropython_embed/` generated by `setup.sh` (gitignored)
**Testing**: Manual verification per spec's acceptance scenarios; build verification via `./build.sh`
**Target Platform**: ESP32-S3-MINI-1, XIAO form factor; Arduino ESP32 core 3.x on ESP-IDF 5.x
**Project Type**: Embedded firmware — scripting runtime component
**Performance Goals**: GC pause ≤ 20ms; app load ≤ 2s from menu select; emergency escape ≤ 1s
**Constraints**: ≤ 128 KiB DRAM for MicroPython heap; Core 1 only; no FreeRTOS task for VM; no `machine`/`esp32` modules
**Scale/Scope**: Single badge; ~10 apps; interpreter runs one app at a time; soft-reset between launches
