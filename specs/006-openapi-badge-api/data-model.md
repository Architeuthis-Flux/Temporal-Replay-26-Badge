## State Transitions

`BoopStatus` drives the pairing state machine in `BadgeIR`:

```
UNKNOWN (parse error — retry)
PENDING ──── poll getBoopStatus ──── CONFIRMED → enter PAIRED state
         └── timeout / button ────── NOT_REQUESTED → reset to IDLE
```

`ProbeResult` drives the badge activation polling in `osUnpairedFlow`:

```
ok=false, httpCode=-1   → "conn err" display status
ok=false, httpCode=404  → "HTTP 404" display status (badge not yet registered)
ok=true,  httpCode=200  → trigger fetchBadgeXBM → enter PAIRED state
```

## Validation Rules

- All char arrays are null-terminated and truncated to their declared size minus 1.
- On `ok == false`, all fields beyond `ok` and `httpCode` are zero-initialized.
- `pairingId == 0` means "not set" (backend IDs start at 1).
- `PingRecord.data` truncation: if the backend JSON exceeds `BADGE_PING_DATA_MAX - 1`
  bytes, the buffer is filled to capacity and null-terminated. No byte-count field is
  needed because apps using pings should design their `data` payloads to fit; the firmware
  is a passthrough, not a JSON processor.
- `GetPingsResult.count > BADGE_PINGS_MAX_RECORDS` cannot occur — excess backend records
  are silently discarded. Use `next_cursor` to page.
