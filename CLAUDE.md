# Temporal Badge ŌĆö Development Guidelines

Conference badge (ESP32-S3-MINI-1 / XIAO form factor). Firmware is Arduino C++ (PlatformIO) with an embedded MicroPython runtime. All active source is in `firmware/src/`. Build via `firmware/build.sh`; flash via `ignition/start.sh`.

---

## Tech Stack

- **Language**: Arduino C++ (C++17, ESP32 Arduino core 3.x)
- **Scripting**: MicroPython v1.27.0 (embed port, generated by `setup.sh`)
- **Display**: U8G2 ŌĆö `U8G2_SSD1306_128X64_NONAME_F_HW_I2C`
- **IR**: IRremote, NEC protocol (TCP-IR handshake protocol, spec-008)
- **Persistence**: `Preferences` (ESP32 NVS)
- **HTTP/JSON**: `HTTPClient` + `ArduinoJson` 6.21.5
- **QR**: libqrcode embedded (`qrcode.c/h`)
- **Flash layout**: 4 MB factory app + 4 MB LittleFS (apps at 0x670000)

## Commands

### Build (`firmware/`)

```bash
# First-time setup ŌĆö installs PlatformIO, Temporal CLI, Python deps
cd ignition && ./setup.sh

# Build firmware (prompts for WiFi/server config ŌåÆ writes settings.txt)
cd firmware && ./build.sh

# Build a specific environment without prompts (settings.txt already configured)
cd firmware && ./build.sh echo -n
```

**Config**: edit `firmware/settings.txt` with WiFi SSID/password/server URL (gitignored). Copy from `firmware/settings.txt.example` on a fresh checkout.

### Flash (`ignition/`)

```bash
# Build delta + flash the currently connected badge batch
cd ignition && ./start.sh

# Flash a specific environment
cd ignition && ./start.sh -e echo

# Skip build, flash last compiled binary
cd ignition && ./start.sh --no-build

# Build only, do not flash
cd ignition && ./start.sh --build-only

# View live workflow status and logs
open http://localhost:8233
```

Badges are reset into bootloader automatically via USB serial ŌĆö no button sequence needed.
Each run is one operator-confirmed hub batch. Ignition starts one
`FlashBadgesWorkflow` parent and one human-readable `BadgeFlashWorkflow` child
per detected badge. Unplug the finished hub, connect the next hub, and start a
new Ignition run for the next batch.

### Hardware harness plan (`Temporal-Badge/`)

Two-badge firmware hardware testing is planned under
`specs/012-hardware-test-harness/`. Read
`docs/hardware-test-harness.md` before implementing or running it. Key points:

- Use Ignition as the default multi-flash path: `cd ignition && ./start.sh -e echo-dev`.
- Close serial monitors/readers before flashing; re-detect ports after flashing.
- Store harness logs under `firmware/logs/hardware/` with per-badge raw logs and a merged timeline.
- Use `registrationScanner` only for backend state setup/assertions; backend correctness is covered by its own suite.
- Physical boop tests must use real UI button injection (`badge.dev("btn", ...)`) and real IR, not a mocked pairing row.

### Serial logging (`firmware/`)

```bash
cd firmware && python3 serial_log.py
```

For spec-010 hardware testing, enable focused serial categories in `settings.txt`:
`log_notify = 1` and/or `log_zigmoji = 1`.

**Kill serial before flashing** ŌĆö `serial_log.py` and the flash worker compete for the same USB port. `pkill -f serial_log.py` first.

### PlatformIO directly

If running `pio` manually, always use the PlatformIO-bundled binary to avoid Python environment conflicts:

```bash
~/.platformio/penv/bin/pio run -e echo
```

The system `pio` shim (pyenv) may resolve to the wrong Python and fail with fatfs import errors.

## Module Reference

`firmware/src/` is organized by domain. See `firmware/src/README.md` for
the full folder map + "how do IŌĆ”" recipe table; this CLAUDE.md table is
the high-signal index.

| Folder | Owns |
|--------|------|
| `api/` | `BadgeAPI` HTTP transport (HMAC auth, all REST calls), `BadgeAPI_types.h` (generated from `specs/openapi.json`), `WiFiService` (connect + sync trigger + ping poll) |
| `boops/` | Boop pairing ŌĆö `BadgeBoops.h` facade + `BoopsJournal` (`/boops.json`) + `BoopsProtocol` (state machine + frame codec) + `BoopsHandlers` (peer / exhibit / queue / kiosk / checkin) + `BoopsFeedback` (marquee + cues) |
| `hardware/` | Drivers ŌĆö `oled` (U8G2), `LEDmatrix`, `Inputs`, `IMU`, `Haptics`, `Power`, `qrcode`, board-specific `*Defines.h`, `HardwareConfig.h` |
| `hw/ir/` | Lower-level NEC encoder/decoder (used by `ir/BadgeIR.cpp`) |
| `identity/` | Per-badge state ŌĆö `BadgeUID` (eFuse + UUID v5), `BadgeStorage` (NVS), `BadgeInfo` (`/badgeInfo.txt`), `BadgePairing` (QR enrollment), `BadgeQR`, `BadgeVersion` |
| `infra/` | Process plumbing ŌĆö `Scheduler`, `Filesystem` (atomic writes), `BadgeConfig` (settings + font-family table), `Bitops`, `DebugLog` |
| `ir/` | High-level IR transport ŌĆö `BadgeIR` (Core 0 task, Python queue, multi-word frames for boop pairing) |
| `messaging/` | Messages + zigmojis ŌĆö `MessageInbox`, `ZigmojiInbox`, `SendMessage`, `BadgePendingBoopApi` (offline boop reconciler), `BadgeJournal` (atomic-write helper) |
| `micropython/` | C++ side of the Python bridge ŌĆö `ReplayMicropythonAPI.cpp` (pump aggregator), `MicroPythonBridge.cpp`, `StartupFiles*`, `badge_mp_api/` subdir with per-subsystem HAL impls (`mp_api_display.cpp`, etc.) |
| `screens/` | GUI screen TUs ŌĆö base class `Screen`, one TU per screen, `EmojiText` + `TextInputLayouts` shared helpers |
| `ui/` | Display rendering above hardware ŌĆö `GUI` (GUIManager + screen stack), `BadgeDisplay`, `FontCatalog` (u8g2 font tables), `Images.h`, `graphics.h` |

The Python-side MicroPython usermod (QSTR table + `MP_DEFINE_CONST_FUN_OBJ_*`)
lives outside `firmware/src/` at `firmware/micropython/usermods/temporalbadge/`.
`main.cpp` and `BadgeGlobals.cpp` stay at `firmware/src/` root.

## Gotchas

- **Kill serial before flashing** ŌĆö `serial_log.py` and the ignition flash worker compete for the same USB port. `pkill -f serial_log.py` first.
- **`settings.txt` is gitignored** ŌĆö new checkouts need to copy from `firmware/settings.txt.example` and fill in WiFi/server credentials before building.
- **`micropython_embed/` is gitignored** ŌĆö must run `cd ignition && ./setup.sh` before first build.
- **After editing `badge_mp/*.c`** ŌĆö the build picks up changes automatically via PlatformIO. No separate regen step needed (setup.sh baked the QSTR tables; source changes trigger a full recompile).
- **Always use `~/.platformio/penv/bin/pio` directly** ŌĆö the system `pio` shim (pyenv) points to the wrong Python and fails with `fatfs` import errors. `build.sh` and `ignition/start.sh` handle this automatically.
- **IR hardware is mode-gated** ŌĆö IR TX/RX enabled only on Boop screen; other modes leave IR off. Don't call IR functions from other modes.
- **Python HTTP does not attach HMAC** ŌĆö `badge.http_post()` is unauthenticated. HMAC is only attached by C-layer `BadgeAPI` calls (`createBoop`, `sendPing`, `getPings`).
- **MicroPython heap from PSRAM** ŌĆö `ps_malloc(128KB)`. If PSRAM unavailable, badge boots without Python support (silent fallback).
- **IR buffer overflow** ŌĆö Python apps polling IR must call `badge.ir_read()` within 50ms or IRremote buffer overflows on Core 0.
- **`BadgeAPI_types.h` is generated** ŌĆö edit `specs/openapi.json` + run `scripts/gen_badge_api_types.py`, don't hand-edit.
- **WPA3 SAE handshake needs ~10s** ŌĆö `WIFI_TIMEOUT_MS = 15000` is intentional; don't reduce it.
- **Main loop stack: 24 KB** ŌĆö bumped for HTTPS/TLS in Messages. Don't reduce.
- **Probe task stack: 8 KB** ŌĆö 4 KB caused stack canary crash with HTTPClient.
- **Ping poll cadence** ŌĆö `notify_poll` defaults to 60s; Messages and Zigmoji push a foreground accelerator that polls every 5s while either app is on the stack.
