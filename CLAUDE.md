# Temporal Badge — Development Guidelines

Conference badge (ESP32-S3-MINI-1 / XIAO form factor). Firmware is Arduino C++ (PlatformIO) with an embedded MicroPython runtime. All active source is in `firmware/src/`. Build via `firmware/build.sh`; flash via `ignition/start.sh`.

---

## Tech Stack

- **Language**: Arduino C++ (C++17, ESP32 Arduino core 3.x)
- **Scripting**: MicroPython v1.27.0 (embed port, generated by `setup.sh`)
- **Display**: U8G2 — `U8G2_SSD1306_128X64_NONAME_F_HW_I2C`
- **IR**: IRremote, NEC protocol (TCP-IR handshake protocol, spec-008)
- **Persistence**: `Preferences` (ESP32 NVS)
- **HTTP/JSON**: `HTTPClient` + `ArduinoJson` 6.21.5
- **QR**: libqrcode embedded (`qrcode.c/h`)
- **Flash layout**: 4 MB factory app + 4 MB LittleFS (apps at 0x670000)

## Commands

### Build (`firmware/`)

```bash
# First-time setup — installs PlatformIO, Temporal CLI, Python deps
cd ignition && ./setup.sh

# Build firmware (prompts for WiFi/server config → writes settings.txt)
cd firmware && ./build.sh

# Build a specific environment without prompts (settings.txt already configured)
cd firmware && ./build.sh echo -n
```

**Config**: edit `firmware/settings.txt` with WiFi SSID/password/server URL (gitignored). Copy from `firmware/settings.txt.example` on a fresh checkout.

### Flash (`ignition/`)

```bash
# Build delta + flash the currently connected badge batch
cd ignition && ./start.sh

# Flash a specific environment
cd ignition && ./start.sh -e echo

# Skip build, flash last compiled binary
cd ignition && ./start.sh --no-build

# Build only, do not flash
cd ignition && ./start.sh --build-only

# View live workflow status and logs
open http://localhost:8233
```

Badges are reset into bootloader automatically via USB serial — no button sequence needed.
Each run is one operator-confirmed hub batch. Ignition starts one
`FlashBadgesWorkflow` parent and one human-readable `BadgeFlashWorkflow` child
per detected badge. Unplug the finished hub, connect the next hub, and start a
new Ignition run for the next batch.

### Hardware harness plan (`Temporal-Badge/`)

Two-badge firmware hardware testing is planned under
`specs/012-hardware-test-harness/`. Read
`docs/hardware-test-harness.md` before implementing or running it. Key points:

- Use Ignition as the default multi-flash path: `cd ignition && ./start.sh -e echo-dev`.
- Close serial monitors/readers before flashing; re-detect ports after flashing.
- Store harness logs under `firmware/logs/hardware/` with per-badge raw logs and a merged timeline.
- Use `registrationScanner` only for backend state setup/assertions; backend correctness is covered by its own suite.
- Physical boop tests must use real UI button injection (`badge.dev("btn", ...)`) and real IR, not a mocked pairing row.

### Serial logging (`firmware/`)

```bash
cd firmware && python3 serial_log.py
```

For spec-010 hardware testing, enable focused serial categories in `settings.txt`:
`log_notify = 1` and/or `log_zigmoji = 1`.

**Kill serial before flashing** — `serial_log.py` and the flash worker compete for the same USB port. `pkill -f serial_log.py` first.

### PlatformIO directly

If running `pio` manually, always use the PlatformIO-bundled binary to avoid Python environment conflicts:

```bash
~/.platformio/penv/bin/pio run -e echo
```

The system `pio` shim (pyenv) may resolve to the wrong Python and fail with fatfs import errors.

## Module Reference

`firmware/src/` is organized by domain. See `firmware/src/README.md` for
the full folder map + "how do I…" recipe table; this CLAUDE.md table is
the high-signal index.

| Folder | Owns |
|--------|------|
| `api/` | `BadgeAPI` HTTP transport (HMAC auth, all REST calls), `BadgeAPI_types.h` (generated from `specs/openapi.json`), `WiFiService` (connect + sync trigger + ping poll) |
| `boops/` | Boop pairing — `BadgeBoops.h` facade + `BoopsJournal` (`/boops.json`) + `BoopsProtocol` (state machine + frame codec) + `BoopsHandlers` (peer / exhibit / queue / kiosk / checkin) + `BoopsFeedback` (marquee + cues) |
| `hardware/` | Drivers — `oled` (U8G2), `LEDmatrix`, `Inputs`, `IMU`, `Haptics`, `Power`, `qrcode`, board-specific `*Defines.h`, `HardwareConfig.h` |
| `hw/ir/` | Lower-level NEC encoder/decoder (used by `ir/BadgeIR.cpp`) |
| `identity/` | Per-badge state — `BadgeUID` (eFuse + UUID v5), `BadgeStorage` (NVS), `BadgeInfo` (`/badgeInfo.txt`), `BadgePairing` (QR enrollment), `BadgeQR`, `BadgeVersion` |
| `infra/` | Process plumbing — `Scheduler`, `Filesystem` (atomic writes), `BadgeConfig` (settings + font-family table), `Bitops`, `DebugLog` |
| `ir/` | High-level IR transport — `BadgeIR` (Core 0 task, Python queue, multi-word frames for boop pairing) |
| `messaging/` | Messages + zigmojis — `MessageInbox`, `ZigmojiInbox`, `SendMessage`, `BadgePendingBoopApi` (offline boop reconciler), `BadgeJournal` (atomic-write helper) |
| `micropython/` | C++ side of the Python bridge — `ReplayMicropythonAPI.cpp` (pump aggregator), `MicroPythonBridge.cpp`, `StartupFiles*`, `badge_mp_api/` subdir with per-subsystem HAL impls (`mp_api_display.cpp`, etc.) |
| `screens/` | GUI screen TUs — base class `Screen`, one TU per screen, `EmojiText` + `TextInputLayouts` shared helpers |
| `ui/` | Display rendering above hardware — `GUI` (GUIManager + screen stack), `BadgeDisplay`, `FontCatalog` (u8g2 font tables), `Images.h`, `graphics.h` |

The Python-side MicroPython usermod (QSTR table + `MP_DEFINE_CONST_FUN_OBJ_*`)
lives outside `firmware/src/` at `firmware/micropython/usermods/temporalbadge/`.
`main.cpp` and `BadgeGlobals.cpp` stay at `firmware/src/` root.

## Menu Structure (spec-010)

```
1. Boop     → IR pairing handshake
2. Zigmoji  → unread zigmoji playback, then send palette
3. Messages → text threads, + New picker, joystick keyboard composer
4. Settings → badge settings
```

The GUI boot path owns the identity gate: paired badges start on the menu,
unpaired badges start on QR and cannot leave QR until paired.

## Style conventions

- **Bitmap arrays should be human-readable.** When writing pixel arrays
  (LED matrix frames, OLED tiles, icon data, font glyphs, sprites…), use
  binary literals `0b00000000` with one literal per row, padded to the row
  width, so the 0s and 1s spatially match the rendered shape. Reading the
  source should reveal the bitmap.
  - For MSB-first formats (`led_set_frame`, OLED `drawXBM`-via-rows, most
    custom blits): write the literal directly — bit 7 is the leftmost
    pixel and the visual is correct.
  - For LSB-first / XBM byte-pair formats (`AppRegistry` icon DATA tuples,
    U8G2 `drawXBM`): the firmware parser reads the byte tuple, but the
    docs/examples should still show a leading visual block of
    `0b<width-bits>` per row so authors can see the shape. Either keep
    the byte tuple for the parser and add a leading visualization
    comment, or generate both from a single source of truth.
  - Never bury bitmaps in opaque hex blobs without a visual companion.
    Hex is fine for the on-disk parser, never for the example you ship
    in docs or comments.

## UI conventions

- **Never use bare button letters in user-facing copy** — neither
  on-device nor in the Jumperless docs site. The Y/B/A/X face buttons
  rotate on nametag flip, and confirm/cancel can be swapped via
  `kSwapConfirmCancel`. A literal "press X" or "A:back" becomes a lie
  depending on the user's settings.
  - **On-device**: route hint / status text through
    `ButtonGlyphs::drawInlineHint(...)` (or the footer helpers, which
    do this internally) — it substitutes glyphs for
    `X / A / B / Y / ^ / v / < / >` and the semantic tokens
    `Confirm / Cancel / Back / Up / Down`. Header banner at
    `ui/ButtonGlyphs.h` has the full token list.
  - **In docs (`Jumperless-docs/docs/badge-developer-guide.md`)**: use
    the SVG glyphs in `Jumperless-docs/docs/img/badge-buttons/`
    (`y.svg`, `b.svg`, `a.svg`, `x.svg`, `lr.svg`, `ud.svg`,
    `all.svg`) — pixel-identical to the on-device cluster art, with
    `<img class="btn" src="img/badge-buttons/y.svg" alt="Y">`. The
    `img.btn` CSS class in `overrides.css` sizes them inline. SVGs
    are regenerated from the firmware bitmaps by
    `firmware/scripts/render_button_glyphs_svg.py` — re-run after
    any change to `ui/ButtonGlyphs.cpp`. Keep the user-facing docs
    free of letters; the position in the diamond is the affordance.

## Code Style

- **Bitmap byte arrays use 8-bit binary literals**, never hex. When a
  `uint8_t[]` / `tuple` / `list` represents a row-major bitmap, write
  each byte as `0b00001111` (always padded to 8 bits) so the dot
  pattern is visible by reading the source. Hex hides the bits.
  - C++17 and MicroPython both accept `0b` literals natively, so this
    works in both `firmware/src/` and `firmware/initial_filesystem/apps/`.
  - The LED matrix and most U8G2 paths are MSB-first (bit 7 = leftmost
    pixel), so the binary literal reads in the natural visual order.
  - U8G2 XBM byte order is LSB-first, so 12×12 `icon.py` `DATA` tuples
    read mirrored — that is acceptable; equivalence at the bit level is
    what matters and `0b00000001` is still strictly the same value as
    `0x01`. Annotate the row's intended visual with a comment when the
    mirror would confuse readers.
  - Bulk sprite/data blobs (e.g. `apps/tardigotchi/sprites.py`,
    `breaksnake/bs_data.py`) stay hex; they're machine-generated and
    not meant for hand-editing.

## Gotchas

- **Kill serial before flashing** — `serial_log.py` and the ignition flash worker compete for the same USB port. `pkill -f serial_log.py` first.
- **`settings.txt` is gitignored** — new checkouts need to copy from `firmware/settings.txt.example` and fill in WiFi/server credentials before building.
- **`micropython_embed/` is gitignored** — must run `cd ignition && ./setup.sh` before first build.
- **After editing `badge_mp/*.c`** — the build picks up changes automatically via PlatformIO. No separate regen step needed (setup.sh baked the QSTR tables; source changes trigger a full recompile).
- **Always use `~/.platformio/penv/bin/pio` directly** — the system `pio` shim (pyenv) points to the wrong Python and fails with `fatfs` import errors. `build.sh` and `ignition/start.sh` handle this automatically.
- **IR hardware is mode-gated** — IR TX/RX enabled only on Boop screen; other modes leave IR off. Don't call IR functions from other modes.
- **Python HTTP does not attach HMAC** — `badge.http_post()` is unauthenticated. HMAC is only attached by C-layer `BadgeAPI` calls (`createBoop`, `sendPing`, `getPings`).
- **MicroPython heap from PSRAM** — `ps_malloc(128KB)`. If PSRAM unavailable, badge boots without Python support (silent fallback).
- **IR buffer overflow** — Python apps polling IR must call `badge.ir_read()` within 50ms or IRremote buffer overflows on Core 0.
- **`BadgeAPI_types.h` is generated** — edit `specs/openapi.json` + run `scripts/gen_badge_api_types.py`, don't hand-edit.
- **WPA3 SAE handshake needs ~10s** — `WIFI_TIMEOUT_MS = 15000` is intentional; don't reduce it.
- **Main loop stack: 24 KB** — bumped for HTTPS/TLS in Messages. Don't reduce.
- **Probe task stack: 8 KB** — 4 KB caused stack canary crash with HTTPClient.
- **Ping poll cadence** — `notify_poll` defaults to 60s; Messages and Zigmoji push a foreground accelerator that polls every 5s while either app is on the stack.
